# Contributing to Pentimento

Thanks for your interest in contributing to Pentimento! This document describes
the development workflow. It is a **developer** workflow, not a normal user
flow.

## Development prerequisites

- Node.js 20+ and npm
- Git 2.20+
- VSCode 1.85+ (for debugging via the Extension Development Host)

## Getting started

```bash
git clone https://example.com/pentimento.git
cd pentimento
npm install
npm run compile      # esbuild bundle -> dist/extension.js
npm run watch        # watch mode
npm run lint
npm run test:unit    # mocha unit tests
npm run test         # @vscode/test-electron integration tests
```

Press <kbd>F5</kbd> in VSCode to launch an Extension Development Host with
Pentimento loaded.

## Architecture & boundaries

Read `docs/TECHNICAL_DESIGN.md` before making non-trivial changes. Hard rules:

- **No external patch files.** Never add `.patch` / `.diff` import or parsing.
  `PatchSelectionType` must never include `'patch-file'`.
- **No Diff Editor / WebView code viewer.** Use `createTextEditorDecorationType`
  + `setDecorations` only.
- **No terminal operations for users.** Git runs only via `GitRunner` with
  argument arrays (never shell strings).
- **Accuracy over coverage.** Never apply historical patch line numbers to the
  current HEAD. Mark uncertain lines as ambiguous.
- **Multi-patch from day one.** No single global `activePatch`.
- **Coexist with GitLens and other extensions.** Do not replace hovers, modify
  configs, or touch other extensions' decorations.

## Commit message convention

Follow OpenStack / Gerrit style:

- First line: imperative, <= 50 chars, no trailing period, no `feat:`/`fix:`
  prefix.
- Blank line, then a body wrapped at <= 72 chars explaining why and what
  changed, and compatibility impact.
- Add footers (`Closes-Bug: #...`, `Related-Task: #...`, `Depends-On: I...`)
  as needed. `Change-Id` is generated by the commit-msg hook; do not hand-write
  it.

## Testing

- Add unit tests for parsers and pure logic (`test/unit/`).
- Add integration tests with real temporary Git repositories for end-to-end
  behavior (`test/integration/`).
- Update `docs/IMPLEMENTATION_STATUS.md` for any feature you add or change.
  Do not mark a feature "已完成/completed" unless code + tests + docs are done.

## Pull requests

- Keep PRs focused; split unrelated changes.
- Ensure `npm run lint`, `npm run check`, `npm run test:unit` pass.
- Describe the change, the rationale, and the test coverage.

By contributing, you agree to abide by the [Code of Conduct](./CODE_OF_CONDUCT.md).
